# Профілювання та оптимізація продуктивності

Цей документ містить результати профілювання, аналізу "гарячих точок" (bottlenecks) та застосованих оптимізацій для консольного математичного додатка `BachelorProject.exe`.

## 1. Методологія та інструменти
* **Інструмент профілювання:** Вбудований профайлер Microsoft Visual Studio (Performance Profiler -> CPU Usage & Memory Usage).
* **Метрики продуктивності:** Час виконання основного циклу Монте-Карло (мілісекунди), процесорний час (%).
* **Тестові набори даних (+2 бали):**
  * **Малий:** `NN = 10,000` ітерацій.
  * **Середній:** `NN = 100,000` ітерацій.
  * **Великий:** `NN = 1,000,000` ітерацій.

## 2. Базове профілювання (До оптимізації)
Під час запуску профайлера (CPU Usage) у режимі Debug на великому наборі даних (`NN = 1,000,000`) було визначено "критичний шлях" виконання програми. Виявлено три головні "гарячі точки" (Hot spots):

1. **Генерація псевдовипадкових чисел (`[Системный код] ucrtbased.dll` / `rand()`):** Системні виклики для отримання випадкових значень споживали найбільше ресурсів — **38,85%** загального процесорного часу.
2. **Функція згортки (`svertka`):** Математичні обчислення в циклі забирали **28,62%** часу.
3. **Операції зсуву масивів у пам'яті (Власний час функції `main`):** Поелементний зсув трьох масивів розмірністю 60 через звичайний цикл `for` споживав **25,71%** часу через відсутність ефективної роботи з блоками пам'яті.

Базовий час виконання для 1,000,000 ітерацій становив **567.79 ms**.

## 3. Застосовані оптимізації (Глибокі оптимізації +5 балів)
Для усунення вузьких місць було переписано вихідний код та змінено налаштування компілятора:
1. **Алгоритмічна оптимізація (Генератор RNG):** Застарілу конструкцію з 12 викликів `rand()` замінено на апаратний генератор `std::mt19937` у комбінації з `std::normal_distribution`. Це математично точніше і швидше.
2. **Оптимізація роботи з пам'яттю:** Поелементний зсув масивів через цикл `for` замінено на функцію `std::memmove()`. Вона виконує копіювання суцільних блоків пам'яті на рівні інструкцій процесора (векторизація).
3. **APM Моніторинг (+3 бали):** Імплементовано бібліотеку `<chrono>` для постійного заміру часу виконання основного циклу. Метрики автоматично виводяться в консоль та логуються у файл `tpar.rez`.
4. **Оптимізація компілятора та розміру бінарника (+2 бали):** Активовано прапорець компілятора `/O2` (Maximize Speed) у конфігурації Release та вимкнено генерацію налагоджувальних символів.

## 4. Результати після оптимізації
Нижче наведено порівняння часу виконання до і після застосування оптимізацій (дані отримані через вбудований APM-моніторинг):

| Обсяг даних (NN) | До оптимізації (Debug/Base) | Після оптимізації (Release/O2) | Відсоток покращення |
| :--- | :--- | :--- | :--- |
| 10,000 (Малий) | ~ 5.60 ms | **1.55 ms** | **~72.3%** |
| 100,000 (Середній) | ~ 56.70 ms | **14.70 ms** | **~74.0%** |
| 1,000,000 (Великий) | 567.79 ms | **146.81 ms** | **74.1%** |

**Висновок:** Завдяки зміні структури роботи з пам'яттю (`memmove`), переходу на сучасний алгоритм генерації шуму та агресивним оптимізаціям компілятора (/O2), загальну швидкодію застосунку вдалося підвищити на понад 74%. Застосунок успішно масштабується і виконує обчислення з надзвичайно високою ефективністю (понад 6,8 млн ітерацій за секунду).
